🔧 Redis対応Magic Link管理システム

🆕 Redis一本化: クラス変数の問題を解決し、永続的で信頼性の高いトークン管理を実現

🔄 Magic Link認証フロー（Redis対応版）

📧 1. Magic Linkリクエスト

  1. ユーザーがメールアドレスを入力
  2. サーバーがトークンを生成（32文字のランダム文字列）
  3. Redisにトークンを保存（キー: magic_link:{token_hash}）
  4. TTL: 24時間（1440分）で自動削除
  5. 開発環境ではトークンを直接返却

🔐 2. Magic Link検証

  1. ユーザーがトークンを入力
  2. サーバーがトークンをハッシュ化
  3. Redisからトークンデータを取得
  4. 有効期限・使用済みチェック
  5. 検証成功時、トークンを使用済みにマーク
  6. JWTトークンを生成・返却

💾 Redisデータ構造

キー形式

magic_link:{token_hash}

保存データ（JSON形式）

{
    "email": "test@example.com",
    "token": "03dd341054f4436e9669...",
    "expires_at": "2025-08-22T12:00:00",
    "used": false,
    "created_at": "2025-08-22T11:30:00"
}

TTL管理

  • 有効期限: 24時間（1440分）
  • 自動削除: RedisのTTLで期限切れトークンを自動クリーンアップ
  • 永続性: サーバー再起動後もデータ保持

🔍 従来のクラス変数方式との違い

項目 従来方式（クラス変数） 新方式（Redis）
データ永続性 ❌ サーバー再起動で消失 ✅ 永続的に保持
インスタンス間共有 ❌ リクエストごとにリセット ✅ 全インスタンスで共有
スケーラビリティ ❌ 単一サーバーでのみ有効 ✅ マルチサーバー対応
自動クリーンアップ ❌ 手動管理が必要 ✅ TTLで自動削除

🎫 JWT管理システム（DB連携版）

🆕 完全DB連携: JWTトークンの発行・検証・管理をデータベースで一元管理

🔐 JWT認証フロー

1. 認証成功時

  1. JWTアクセストークン生成（有効期限: 15分）
  2. リフレッシュトークン生成（有効期限: 30日）
  3. DBにセッション情報を保存
  4. Redisにセッションキャッシュを保存

2. アクセストークン更新

  1. リフレッシュトークンで検証
  2. 新しいアクセストークンを生成
  3. セッション情報を更新

3. ログアウト時

  1. JWTブラックリストに追加
  2. セッションを無効化
  3. Redisキャッシュを削除

💾 データベーステーブル構成

sessions（セッション管理）

  • session_id: セッション一意識別子
  • user_id: ユーザーID
  • device_id: 端末識別
  • is_revoked: 失効フラグ

refresh_tokens（リフレッシュトークン）

  • token_id: トークン一意識別子
  • session_id: セッションとの紐付け
  • expires_at: 有効期限
  • is_revoked: 失効フラグ

jwt_blacklist（JWT即時失効）

  • token_hash: ハッシュ化されたJWT
  • expires_at: 失効日時
  • ログアウト時の即座無効化

🔒 セキュリティ機能

🛡️ セッション管理

  • 同時ログイン制限: 端末ごとのセッション管理
  • セッション無効化: 管理者による強制ログアウト
  • 非アクティブタイムアウト: 7日間アクセスなしで自動失効

🚫 不正アクセス対策

  • JWTブラックリスト: ログアウト済みトークンの即座無効化
  • トークンローテーション: 使用時に新しいリフレッシュトークンを発行
  • IPアドレス記録: セッション作成元の追跡

🔄 システム全体の認証フロー

🎯 統合認証システム: Magic Link + JWT + パスワード（任意）の組み合わせで最高レベルのセキュリティを実現

📋 認証方式の選択

🔗 Magic Link認証（推奨）

  • セキュリティ: パスワード不要、ワンタイムトークン
  • 利便性: メールクリックでログイン
  • 管理: Redisで永続的・信頼性の高い管理
  • 有効期限: 24時間（設定可能）

🔐 パスワード認証（オプション）

  • セキュリティ: Argon2idハッシュ、ソルト自動生成
  • 利便性: 従来のログイン方式
  • 管理: データベースで安全に管理
  • 設定: 任意で有効化可能

🔄 認証後のフロー

  1. JWT発行: アクセストークン（15分）+ リフレッシュトークン（30日）
  2. セッション作成: データベースにセッション情報を保存
  3. Redisキャッシュ: 高速アクセス用のセッションキャッシュ
  4. アクセス制御: 各APIエンドポイントでのJWT検証
  5. 自動更新: リフレッシュトークンによる自動トークン更新

💡 開発環境での利点

  • Redis一本化: クラス変数の問題を完全解決
  • 永続性: サーバー再起動後もトークンが保持される
  • デバッグ性: Redis内のトークン状況をリアルタイム確認可能
  • 拡張性: 将来的なマルチサーバー対応も容易
  • 本番環境互換: 開発環境と本番環境で同じ動作
